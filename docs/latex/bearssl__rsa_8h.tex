\subsection{Detailed Description}
\subsection*{R\+SA}

This file documents the R\+SA implementations provided with Bear\+S\+SL. Note that the S\+SL engine accesses these implementations through a configurable A\+PI, so it is possible to, for instance, run a S\+SL server which uses a R\+SA engine which is not based on this code.

\subsubsection*{Key Elements}

R\+SA public and private keys consist in lists of big integers. All such integers are represented with big-\/endian unsigned notation\+: first byte is the most significant, and the value is positive (so there is no dedicated \char`\"{}sign bit\char`\"{}). Public and private key structures thus contain, for each such integer, a pointer to the first value byte ({\ttfamily unsigned char $\ast$}), and a length ({\ttfamily size\+\_\+t}) which is the number of relevant bytes. As a general rule, minimal-\/length encoding is not enforced\+: values may have extra leading bytes of value 0.

R\+SA public keys consist in two integers\+:


\begin{DoxyItemize}
\item the modulus ({\ttfamily n});
\item the public exponent ({\ttfamily e}).
\end{DoxyItemize}

R\+SA private keys, as defined in \href{https://tools.ietf.org/html/rfc3447}{\tt P\+K\+CS\#1}, contain eight integers\+:


\begin{DoxyItemize}
\item the modulus ({\ttfamily n});
\item the public exponent ({\ttfamily e});
\item the private exponent ({\ttfamily d});
\item the first prime factor ({\ttfamily p});
\item the second prime factor ({\ttfamily q});
\item the first reduced exponent ({\ttfamily dp}, which is {\ttfamily d} modulo {\ttfamily p-\/1});
\item the second reduced exponent ({\ttfamily dq}, which is {\ttfamily d} modulo {\ttfamily q-\/1});
\item the C\+RT coefficient ({\ttfamily iq}, the inverse of {\ttfamily q} modulo {\ttfamily p}).
\end{DoxyItemize}

However, the implementations defined in Bear\+S\+SL use only five of these integers\+: {\ttfamily p}, {\ttfamily q}, {\ttfamily dp}, {\ttfamily dq} and {\ttfamily iq}.

\subsubsection*{Security Features and Limitations}

The implementations contained in Bear\+S\+SL have the following limitations and features\+:


\begin{DoxyItemize}
\item They are constant-\/time. This means that the execution time and memory access pattern may depend on the {\itshape lengths} of the private key components, but not on their value, nor on the value of the operand. Note that this property is not achieved through random masking, but \char`\"{}true\char`\"{} constant-\/time code.
\item They support only private keys with two prime factors. R\+SA private keys with three or more prime factors are nominally supported, but rarely used; they may offer faster operations, at the expense of more code and potentially a reduction in security if there are \char`\"{}too many\char`\"{} prime factors.
\item The public exponent may have arbitrary length. Of course, it is a good idea to keep public exponents small, so that public key operations are fast; but, contrary to some widely deployed implementations, Bear\+S\+SL has no problem with public exponents longer than 32 bits.
\item The two prime factors of the modulus need not have the same length (but severely imbalanced factor lengths might reduce security). Similarly, there is no requirement that the first factor ({\ttfamily p}) be greater than the second factor ({\ttfamily q}).
\item Prime factors and modulus must be smaller than a compile-\/time limit. This is made necessary by the use of fixed-\/size stack buffers, and the limit has been adjusted to keep stack usage under 2 kB for the R\+SA operations. Currently, the maximum modulus size is 4096 bits, and the maximum prime factor size is 2080 bits.
\item The R\+SA functions themselves do not enforce lower size limits, except that which is absolutely necessary for the operation to mathematically make sense (e.\+g. a P\+K\+CS\#1 v1.\+5 signature with S\+H\+A-\/1 requires a modulus of at least 361 bits). It is up to users of this code to enforce size limitations when appropriate (e.\+g. the X.\+509 validation engine, by default, rejects R\+SA keys of less than 1017 bits).
\item Within the size constraints expressed above, arbitrary bit lengths are supported. There is no requirement that prime factors or modulus have a size multiple of 8 or 16.
\item When verifying P\+K\+CS\#1 v1.\+5 signatures, both variants of the hash function identifying header (with and without the A\+S\+N.\+1 N\+U\+LL) are supported. When producing such signatures, the variant with the A\+S\+N.\+1 N\+U\+LL is used.
\end{DoxyItemize}

\subsubsection*{Implementations}

Three R\+SA implementations are included\+:


\begin{DoxyItemize}
\item The {\bfseries i32} implementation internally represents big integers as arrays of 32-\/bit integers. It is perfunctory and portable, but not very efficient.
\item The {\bfseries i31} implementation uses 32-\/bit integers, each containing 31 bits worth of integer data. The i31 implementation is somewhat faster than the i32 implementation (the reduced integer size makes carry propagation easier) for a similar code footprint, but uses very slightly larger stack buffers (about 4\% bigger).
\item The {\bfseries i62} implementation is similar to the i31 implementation, except that it internally leverages the 64x64-\/$>$128 multiplication opcode. This implementation is available only on architectures where such an opcode exists. It is much faster than i31.
\item The {\bfseries i15} implementation uses 16-\/bit integers, each containing 15 bits worth of integer data. Multiplication results fit on 32 bits, so this won\textquotesingle{}t use the \char`\"{}widening\char`\"{} multiplication routine on A\+RM Cortex M0/\+M0+, for much better performance and constant-\/time execution. 
\end{DoxyItemize}

Type for R\+SA key pair generator implementation. 

This function generates a new R\+SA key pair whose modulus has bit length {\ttfamily size} bits. The private key elements are written in the {\ttfamily kbuf\+\_\+priv} buffer, and pointer values and length fields to these elements are populated in the provided private key structure {\ttfamily sk}. Similarly, the public key elements are written in {\ttfamily kbuf\+\_\+pub}, with pointers and lengths set in {\ttfamily pk}.

If {\ttfamily pk} is {\ttfamily N\+U\+LL}, then {\ttfamily kbuf\+\_\+pub} may be {\ttfamily N\+U\+LL}, and only the private key is set.

If {\ttfamily pubexp} is not zero, then its value will be used as public exponent. Valid R\+SA public exponent values are odd integers greater than 1. If {\ttfamily pubexp} is zero, then the public exponent will have value 3.

The provided P\+R\+NG ({\ttfamily rng\+\_\+ctx}) must have already been initialized and seeded.

Returned value is 1 on success, 0 on error. An error is reported if the requested range is outside of the supported key sizes, or if an invalid non-\/zero public exponent value is provided. Supported range starts at 512 bits, and up to an implementation-\/defined maximum (by default 4096 bits). Note that key sizes up to 768 bits have been broken in practice, and sizes lower than 2048 bits are usually considered to be weak and should not be used.

Type for a private exponent computing function. 

An R\+SA private key ({\ttfamily \hyperlink{structbr__rsa__private__key}{br\+\_\+rsa\+\_\+private\+\_\+key}}) contains two reduced private exponents, which are sufficient to perform private key operations. However, standard encoding formats for R\+SA private keys require also a copy of the complete private exponent (non-\/reduced), which this function recomputes.

This function suceeds if all the following conditions hold\+:


\begin{DoxyItemize}
\item Both private factors {\ttfamily p} and {\ttfamily q} are equal to 3 modulo 4.
\item The provided public exponent {\ttfamily pubexp} is correct, and, in particular, is odd, relatively prime to {\ttfamily p-\/1} and {\ttfamily q-\/1}, and greater than 1.
\item No internal storage limit is exceeded.
\end{DoxyItemize}

For all private keys produced by the key generator functions ({\ttfamily br\+\_\+rsa\+\_\+keygen} type), this function succeeds. Note that the A\+PI restricts the public exponent to a maximum size of 32 bits.

The encoded private exponent is written in {\ttfamily d} (unsigned big-\/endian convention), and the length (in bytes) is returned. If {\ttfamily d} is {\ttfamily N\+U\+LL}, then the exponent is not written anywhere, but the length is still returned. On error, 0 is returned.

Not all error conditions are detected when {\ttfamily d} is {\ttfamily N\+U\+LL}; therefore, the returned value shall be checked also when actually producing the value.

Type for a R\+SA public key engine. 

The public key engine performs the modular exponentiation of the provided value with the public exponent. The value is modified in place.

The value length ({\ttfamily xlen}) is verified to have {\itshape exactly} the same length as the modulus (actual modulus length, without extra leading zeros in the modulus representation in memory). If the length does not match, then this function returns 0 and {\ttfamily x\mbox{[}\mbox{]}} is unmodified.

It {\ttfamily xlen} is correct, then {\ttfamily x\mbox{[}\mbox{]}} is modified. Returned value is 1 on success, 0 on error. Error conditions include an oversized {\ttfamily x\mbox{[}\mbox{]}} (the array has the same length as the modulus, but the numerical value is not lower than the modulus) and an invalid modulus (e.\+g. an even integer). If an error is reported, then the new contents of {\ttfamily x\mbox{[}\mbox{]}} are unspecified.

R\+SA decryption helper, for S\+S\+L/\+T\+LS. 

This function performs the R\+SA decryption for a R\+S\+A-\/based key exchange in a S\+S\+L/\+T\+LS server. The provided R\+SA engine is used. The {\ttfamily data} parameter points to the value to decrypt, of length {\ttfamily len} bytes. On success, the 48-\/byte pre-\/master secret is copied into {\ttfamily data}, starting at the first byte of that buffer; on error, the contents of {\ttfamily data} become indeterminate.

This function first checks that the provided value length ({\ttfamily len}) is not lower than 59 bytes, and matches the R\+SA modulus length; if neither of this property is met, then this function returns 0 and the buffer is unmodified.

Otherwise, decryption and then padding verification are performed, both in constant-\/time. A decryption error, or a bad padding, or an incorrect decrypted value length are reported with a returned value of 0; on success, 1 is returned. The caller (S\+SL server engine) is supposed to proceed with a random pre-\/master secret in case of error.
