\subsection{Detailed Description}
\subsection*{Hash Functions}

This file documents the A\+PI for hash functions.

\subsubsection*{Procedural A\+PI}

For each implemented hash function, of name \char`\"{}`xxx`\char`\"{}, the following elements are defined\+:


\begin{DoxyItemize}
\item {\ttfamily br\+\_\+xxx\+\_\+vtable}

An externally defined instance of {\ttfamily br\+\_\+hash\+\_\+class}.
\item {\ttfamily br\+\_\+xxx\+\_\+\+S\+I\+ZE}

A macro that evaluates to the output size (in bytes) of the hash function.
\item {\ttfamily br\+\_\+xxx\+\_\+\+ID}

A macro that evaluates to a symbolic identifier for the hash function. Such identifiers are used with H\+M\+AC and signature algorithm implementations.

N\+O\+TE\+: for the \char`\"{}standard\char`\"{} hash functions defined in \href{https://tools.ietf.org/html/rfc5246#section-7.4.1.4.1}{\tt the T\+LS standard}, the symbolic identifiers match the constants used in T\+LS, i.\+e. 1 to 6 for M\+D5, S\+H\+A-\/1, S\+H\+A-\/224, S\+H\+A-\/256, S\+H\+A-\/384 and S\+H\+A-\/512, respectively.
\item {\ttfamily br\+\_\+xxx\+\_\+context}

Context for an ongoing computation. It is allocated by the caller, and a pointer to it is passed to all functions. A context contains no interior pointer, so it can be moved around and cloned (with a simple {\ttfamily memcpy()} or equivalent) in order to capture the function state at some point. Computations that use distinct context structures are independent of each other. The first field of {\ttfamily br\+\_\+xxx\+\_\+context} is always a pointer to the {\ttfamily br\+\_\+xxx\+\_\+vtable} structure; {\ttfamily br\+\_\+xxx\+\_\+init()} sets that pointer.
\item {\ttfamily br\+\_\+xxx\+\_\+init(br\+\_\+xxx\+\_\+context $\ast$ctx)}

Initialise the provided context. Previous contents of the structure are ignored. This calls resets the context to the start of a new hash computation; it also sets the first field of the context structure (called {\ttfamily vtable}) to a pointer to the statically allocated constant {\ttfamily br\+\_\+xxx\+\_\+vtable} structure.
\item {\ttfamily br\+\_\+xxx\+\_\+update(br\+\_\+xxx\+\_\+context $\ast$ctx, const void $\ast$data, size\+\_\+t len)}

Add some more bytes to the hash computation represented by the provided context.
\item {\ttfamily br\+\_\+xxx\+\_\+out(const br\+\_\+xxx\+\_\+context $\ast$ctx, void $\ast$out)}

Complete the hash computation and write the result in the provided buffer. The output buffer M\+U\+ST be large enough to accommodate the result. The context is N\+OT modified by this operation, so this function can be used to get a \char`\"{}partial hash\char`\"{} while still keeping the possibility of adding more bytes to the input.
\item {\ttfamily br\+\_\+xxx\+\_\+state(const br\+\_\+xxx\+\_\+context $\ast$ctx, void $\ast$out)}

Get a copy of the \char`\"{}current state\char`\"{} for the computation so far. For MD functions (M\+D5, S\+H\+A-\/1, S\+H\+A-\/2 family), this is the running state resulting from the processing of the last complete input block. Returned value is the current input length (in bytes).
\item {\ttfamily br\+\_\+xxx\+\_\+set\+\_\+state(br\+\_\+xxx\+\_\+context $\ast$ctx, const void $\ast$stb, uint64\+\_\+t count)}

Set the internal state to the provided values. The \textquotesingle{}stb\textquotesingle{} and \textquotesingle{}count\textquotesingle{} values shall match that which was obtained from {\ttfamily br\+\_\+xxx\+\_\+state()}. This restores the hash state only if the state values were at an appropriate block boundary. This does N\+OT set the {\ttfamily vtable} pointer in the context.
\end{DoxyItemize}

Context structures can be discarded without any explicit deallocation. Hash function implementations are purely software and don\textquotesingle{}t reserve any resources outside of the context structure itself.

\subsubsection*{Object-\/\+Oriented A\+PI}

For each hash function that follows the procedural A\+PI described above, an object-\/oriented A\+PI is also provided. In that A\+PI, function pointers from the vtable ({\ttfamily br\+\_\+xxx\+\_\+vtable}) are used. The vtable incarnates object-\/oriented programming. An introduction on the O\+OP concept used here can be read on the Bear\+S\+SL Web site\+:~\newline
 ~~~\href{https://www.bearssl.org/oop.html}{\tt https\+://www.\+bearssl.\+org/oop.\+html}

The vtable offers functions called {\ttfamily init()}, {\ttfamily update()}, {\ttfamily out()}, {\ttfamily set()} and {\ttfamily set\+\_\+state()}, which are in fact the functions from the procedural A\+PI. That vtable also contains two informative fields\+:


\begin{DoxyItemize}
\item {\ttfamily context\+\_\+size}

The size of the context structure ({\ttfamily br\+\_\+xxx\+\_\+context}), in bytes. This can be used by generic implementations to perform dynamic context allocation.
\item {\ttfamily desc}

A \char`\"{}descriptor\char`\"{} field that encodes some information on the hash function\+: symbolic identifier, output size, state size, internal block size, details on the padding.
\end{DoxyItemize}

Users of this object-\/oriented A\+PI (in particular generic H\+M\+AC implementations) may make the following assumptions\+:


\begin{DoxyItemize}
\item Hash output size is no more than 64 bytes.
\item Hash internal state size is no more than 64 bytes.
\item Internal block size is a power of two, no less than 16 and no more than 256.
\end{DoxyItemize}

\subsubsection*{Multi-\/\+Hasher}

An aggregate hasher is provided, that can compute several standard hash functions in parallel. It uses {\ttfamily \hyperlink{structbr__multihash__context}{br\+\_\+multihash\+\_\+context}} and a procedural A\+PI. It is configured with the implementations (the vtables) that it should use; it will then compute all these hash functions in parallel, on the same input. It is meant to be used in cases when the hash of an object will be used, but the exact hash function is not known yet (typically, streamed processing on X.\+509 certificates).

Only the standard hash functions (M\+D5, S\+H\+A-\/1, S\+H\+A-\/224, S\+H\+A-\/256, S\+H\+A-\/384 and S\+H\+A-\/512) are supported by the multi-\/hasher.

\subsubsection*{G\+H\+A\+SH}

G\+H\+A\+SH is not a generic hash function; it is a {\itshape universal} hash function, which, as the name does not say, means that it C\+A\+N\+N\+OT be used in most places where a hash function is needed. G\+H\+A\+SH is used within the G\+CM encryption mode, to provide the checked integrity functionality.

A G\+H\+A\+SH implementation is basically a function that uses the type defined in this file under the name {\ttfamily br\+\_\+ghash}\+: \begin{DoxyVerb}typedef void (*br_ghash)(void *y, const void *h, const void *data, size_t len);
\end{DoxyVerb}


The {\ttfamily y} pointer refers to a 16-\/byte value which is used as input, and receives the output of the G\+H\+A\+SH invocation. {\ttfamily h} is a 16-\/byte secret value (that serves as key). {\ttfamily data} and {\ttfamily len} define the input data.

Three G\+H\+A\+SH implementations are provided, all constant-\/time, based on the use of integer multiplications with appropriate masking to cancel carry propagation.

\subsubsection{\texorpdfstring{br\+\_\+ghash}{br\_ghash}}
{\footnotesize\ttfamily typedef void($\ast$ br\+\_\+ghash) (void $\ast$y, const void $\ast$h, const void $\ast$data, size\+\_\+t len)}



Type for a G\+H\+A\+SH implementation. 

G\+H\+A\+SH is a sort of keyed hash meant to be used to implement G\+CM in combination with a block cipher (with 16-\/byte blocks).

The {\ttfamily y} array has length 16 bytes and is used for input and output; in a complete G\+H\+A\+SH run, it starts with an all-\/zero value. {\ttfamily h} is a 16-\/byte value that serves as key (it is derived from the encryption key in G\+CM, using the block cipher). The data length ({\ttfamily len}) is expressed in bytes. The {\ttfamily y} array is updated.

If the data length is not a multiple of 16, then the data is implicitly padded with zeros up to the next multiple of 16. Thus, when using G\+H\+A\+SH in G\+CM, this method may be called twice, for the associated data and for the ciphertext, respectively; the zero-\/padding implements exactly the G\+CM rules.
