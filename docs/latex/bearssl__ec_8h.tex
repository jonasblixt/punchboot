\subsection{Detailed Description}
\subsection*{Elliptic Curves}

This file documents the EC implementations provided with Bear\+S\+SL, and E\+C\+D\+SA.

\subsubsection*{Elliptic Curve A\+PI}

Only \char`\"{}named curves\char`\"{} are supported. Each EC implementation supports one or several named curves, identified by symbolic identifiers. These identifiers are small integers, that correspond to the values registered by the \href{http://www.iana.org/assignments/tls-parameters/tls-parameters.xhtml#tls-parameters-8}{\tt I\+A\+NA}.

Since all currently defined elliptic curve identifiers are in the 0..31 range, it is convenient to encode support of some curves in a 32-\/bit word, such that bit x corresponds to curve of identifier x.

An EC implementation is incarnated by a {\ttfamily \hyperlink{structbr__ec__impl}{br\+\_\+ec\+\_\+impl}} instance, that offers the following fields\+:


\begin{DoxyItemize}
\item {\ttfamily supported\+\_\+curves}

A 32-\/bit word that documents the identifiers of the curves supported by this implementation.
\item {\ttfamily generator()}

Callback method that returns a pointer to the conventional generator point for that curve.
\item {\ttfamily order()}

Callback method that returns a pointer to the subgroup order for that curve. That value uses unsigned big-\/endian encoding.
\item {\ttfamily xoff()}

Callback method that returns the offset and length of the X coordinate in an encoded point.
\item {\ttfamily mul()}

Multiply a curve point with an integer.
\item {\ttfamily mulgen()}

Multiply the curve generator with an integer. This may be faster than the generic {\ttfamily mul()}.
\item {\ttfamily muladd()}

Multiply two curve points by two integers, and return the sum of the two products.
\end{DoxyItemize}

All curve points are represented in uncompressed format. The {\ttfamily mul()} and {\ttfamily muladd()} methods take care to validate that the provided points are really part of the relevant curve subgroup.

For all point multiplication functions, the following holds\+:


\begin{DoxyItemize}
\item Functions validate that the provided points are valid members of the relevant curve subgroup. An error is reported if that is not the case.
\item Processing is constant-\/time, even if the point operands are not valid. This holds for both the source and resulting points, and the multipliers (integers). Only the byte length of the provided multiplier arrays (not their actual value length in bits) may leak through timing-\/based side channels.
\item The multipliers (integers) M\+U\+ST be lower than the subgroup order. If this property is not met, then the result is indeterminate, but an error value is not ncessearily returned.
\end{DoxyItemize}

\subsubsection*{E\+C\+D\+SA}

E\+C\+D\+SA signatures have two standard formats, called \char`\"{}raw\char`\"{} and \char`\"{}asn1\char`\"{}. Internally, such a signature is a pair of modular integers {\ttfamily (r,s)}. The \char`\"{}raw\char`\"{} format is the concatenation of the unsigned big-\/endian encodings of these two integers, possibly left-\/padded with zeros so that they have the same encoded length. The \char`\"{}asn1\char`\"{} format is the D\+ER encoding of an A\+S\+N.\+1 structure that contains the two integer values\+: \begin{DoxyVerb}ECDSASignature ::= SEQUENCE {
    r   INTEGER,
    s   INTEGER
}
\end{DoxyVerb}


In general, in all of X.\+509 and S\+S\+L/\+T\+LS, the \char`\"{}asn1\char`\"{} format is used. Bear\+S\+SL offers E\+C\+D\+SA implementations for both formats; conversion functions between the two formats are also provided. Conversion of a \char`\"{}raw\char`\"{} format signature into \char`\"{}asn1\char`\"{} may enlarge a signature by no more than 9 bytes for all supported curves; conversely, conversion of an \char`\"{}asn1\char`\"{} signature to \char`\"{}raw\char`\"{} may expand the signature but the \char`\"{}raw\char`\"{} length will never be more than twice the length of the \char`\"{}asn1\char`\"{} length (and usually it will be shorter).

Note that for a given signature, the \char`\"{}raw\char`\"{} format is not fully deterministic, in that it does not enforce a minimal common length.

Compute EC public key from EC private key. 

This function uses the provided elliptic curve implementation ({\ttfamily impl}) to compute the public key corresponding to the private key held in {\ttfamily sk}. The public key point is written into {\ttfamily kbuf}, which is then linked from the {\ttfamily $\ast$pk} structure. The size of the public key point, i.\+e. the number of bytes used in {\ttfamily kbuf}, is returned.

If {\ttfamily kbuf} is {\ttfamily N\+U\+LL}, then the public key point is N\+OT computed, and the public key structure {\ttfamily $\ast$pk} is unmodified ({\ttfamily pk} may be {\ttfamily N\+U\+LL} in that case). The size of the public key point is still returned.

If {\ttfamily pk} is {\ttfamily N\+U\+LL} but {\ttfamily kbuf} is not {\ttfamily N\+U\+LL}, then the public key point is computed and stored in {\ttfamily kbuf}, and its size is returned.

If the curve used by the private key is not supported by the curve implementation, then this function returns zero.

The private key M\+U\+ST be valid. An off-\/range private key value is not necessarily detected, and leads to unpredictable results.

Generate a new EC private key. 

If the specified {\ttfamily curve} is not supported by the elliptic curve implementation ({\ttfamily impl}), then this function returns zero.

The {\ttfamily sk} structure fields are set to the new private key data. In particular, {\ttfamily sk.\+x} is made to point to the provided key buffer ({\ttfamily kbuf}), in which the actual private key data is written. That buffer is assumed to be large enough. The {\ttfamily B\+R\+\_\+\+E\+C\+\_\+\+K\+B\+U\+F\+\_\+\+P\+R\+I\+V\+\_\+\+M\+A\+X\+\_\+\+S\+I\+ZE} defines the maximum size for all supported curves.

The number of bytes used in {\ttfamily kbuf} is returned. If {\ttfamily kbuf} is {\ttfamily N\+U\+LL}, then the private key is not actually generated, and {\ttfamily sk} may also be {\ttfamily N\+U\+LL}; the minimum length for {\ttfamily kbuf} is still computed and returned.

If {\ttfamily sk} is {\ttfamily N\+U\+LL} but {\ttfamily kbuf} is not {\ttfamily N\+U\+LL}, then the private key is still generated and stored in {\ttfamily kbuf}.
